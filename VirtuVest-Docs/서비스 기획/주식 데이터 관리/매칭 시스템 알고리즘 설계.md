# 주문 프로세스
---
1. 체결 API 에서 체결가와 체결 수량을 계속 받음. ([한투 체결 API](https://apiportal.koreainvestment.com/apiservice-apiservice?/uapi/domestic-stock/v1/quotations/inquire-ccnl))
2. 내 서비스상의 주문들은 매수/매도 호가별 FIFO 구조(이하 주문요청큐)로 관리됨.
     (`Map<호가, 매수주문요청큐>`, `Map<호가, 매도주문요청큐>`)
3. 만약 내 서비스에서 매수/매도 주문이 들어오면, 우선 pending 상태로 알맞는 주문요청큐에 넣음
4. 체결 API 의 틱 데이터 수신하여, 체결가 확인.
5. 해당 체결가 보다 크거나 같은 매수 호가 중, 가장 호가가 높은 큐에서 매수 주문을 선택.
     `min(API 체결 수량, 매수 주문 수량)` 만큼 체결 처리.
6. 해당 체결가 보다 작거나 같은 매도 호가 중, 가장 호가가 낮은 큐에서 매도 주문을 선택.
     `min(API 체결 수량, 2번 매수 주문 수량)` 만큼 체결 처리.
     (5번과 6번은 함께 처리된다. 즉, 동일한 체결 틱 데이터를 사용.)
7. 선택한 주문의 수량 < API 체결 수량 :
     주문요청큐에서 해당 주문을 dequeue하고 `EXECUTED` 로 업데이트. 남은 API 체결 수량을 가지고 4번부터 재시작
8. 선택한 주문의 수량 = API 체결 수량 :
     주문요청큐에서 해당 주문을 dequeue하고 `EXECUTED` 로 업데이트. 다음 API 체결 응답 받아 4번부터 재시작
9. 선택한 주문의 수량 > API 체결 수량 :
     `PARTIALLY_FILLED` 로 업데이트. 4번부터 재시작 (dequeue 안했으므로 재처리됨.)
