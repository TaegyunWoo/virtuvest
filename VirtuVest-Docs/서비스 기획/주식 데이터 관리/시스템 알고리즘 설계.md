# 주문 매칭 프로세스
---
1. 체결 API 에서 체결가와 체결 수량을 계속 받음. ([한투 실시간 체결가 조회 API](https://apiportal.koreainvestment.com/apiservice-apiservice?/tryitout/H0STCNT0))
2. 내 서비스상의 주문들은 매수/매도 호가별 FIFO 구조(이하 주문요청큐)로 관리됨.
	- 지정가 주문 큐 : `Map<호가, 매수주문요청큐>`, `Map<호가, 매도주문요청큐>`
	- 시장가 주문 큐 : `Deque<매수주문요청큐>`, `Deque<매도주문요청큐>`
3. 만약 내 서비스에서 매수/매도 주문이 들어오면, 우선 pending 상태로 알맞는 주문요청큐에 넣음
4. 체결 API 의 틱 데이터 수신하여, 체결가 확인.
5. 매수 주문 처리 : 수신받은  틱데이터의 종목의 주문큐에서 고른다.
	1. 체결 처리할 매수 주문 선택
		1. 시장가 매수 주문 큐에 주문이 존재한다면: 해당 주문을 선택
		2. 시장가 매수 주문 큐에 주문이 존재하지 않는다면: 4번의 체결가 보다 크거나 같은 매수 호가 중, 가장 호가가 높은 큐에서 매수 주문을 선택.
	2. `min(API 체결 수량, 매수 주문 수량)` 만큼 체결 처리.
6. 매도 주문 처리 (5번과 6번은 함께 처리된다. 즉, 동일한 체결 틱 데이터를 사용.)
	1. 체결 처리할 매도 주문 선택
		1. 시장가 매도 주문 큐에 주문이 존재한다면: 해당 주문을 선택
		2. 시장가 매도 주문 큐에 주문이 존재하지 않는다면: 4번의 체결가 보다 작거나 같은 매도 호가 중, 가장 호가가 낮은 큐에서 매수 주문을 선택.
	2. `min(API 체결 수량, 2번 매도 주문 수량)` 만큼 체결 처리.
7. 선택한 주문의 수량 < API 체결 수량 :
     주문요청큐에서 해당 주문을 dequeue하고 `EXECUTED` 로 업데이트. 남은 API 체결 수량을 가지고 4번부터 재시작
8. 선택한 주문의 수량 = API 체결 수량 :
     주문요청큐에서 해당 주문을 dequeue하고 `EXECUTED` 로 업데이트. 다음 API 체결 응답 받아 4번부터 재시작
9. 선택한 주문의 수량 > API 체결 수량 :
     `PARTIALLY_FILLED` 로 업데이트. 5번부터 재시작 (dequeue 안했으므로 재처리됨.)

# 체결가 구독 종목 관리
---
> https://gemini.google.com/app/427b11fc133b2145?hl=ko
- 사용 API : [한투 실시간 체결가 조회 API](https://apiportal.koreainvestment.com/apiservice-apiservice?/tryitout/H0STCNT0)
- 한 세션에 50개 종목까지 구독 가능

## 1) 종목 선정 우선순위

| 우선순위 | 대상                |
| ---- | ----------------- |
| 1순위  | 미체결 주문이 있는 종목     |
| 2순위  | 현재 사용자들이 보고 있는 종목 |
| 3순위  | 거래량 상위 20위 종목     |
- 각 대상마다의 세부 우선순위는 거래량으로 설정한다.
	- e.g.) 미체결 주문이 있는 종목들을 거래량이 높은 순으로 정렬
## 2) 세션 관리
- 구독해야하는 종목의 수에 따라 세션을 최대 5개까지 연결할 수 있다.
	- 세션 하나당 최대 50개 종목까지 구독 허용
- 현재 연결한 세션 수 = `Min(5 , (구독해야하는 종목 수 / 50) + 1)`
- Redis에서 현재 구독중인 종목을 세션별로 관리한다.

| 세션 1            | 세션 2             | 세션 3              | 세션 4              | 세션 5              |
| --------------- | ---------------- | ----------------- | ----------------- | ----------------- |
| - 종목 1<br>- ... | - 종목 51<br>- ... | - 종목 101<br>- ... | - 종목 151<br>- ... | - 종목 201<br>- ... |

## 3) Fallback 전략
- 우선순위에 따라 구독해야하는 종목의 수가 250개(최대구독수 * 세션수)를 넘는다면, 나머지는 Rest API 호출로 커버한다.
	- [주식 현재가 체결 Rest API](https://apiportal.koreainvestment.com/apiservice-apiservice?/uapi/domestic-stock/v1/quotations/inquire-ccnl)
- 0.2sec 마다 호출한다.
# 시스템 플로우
---
> 모든 주문처리는 RDB가 아닌 **==Redis를 기반==** 으로 한다. RDB는 데이터 백업 및 팔로워 역할을 수행한다.
## 1) 주문 시스템 플로우
1. **Front -> 주문 서버에 주문 요청**
	- 매수/매도, 시장가/지정가 주문
2. **주문 서버 : 주문 요청 처리**
	1. Redis 에 `주문 데이터` 저장 (`ORDER_REQUEST` 테이블과 동일한 데이터구조로)
	2. Kafka 에 주문 요청 이벤트 발행
3. **주문 요청 이벤트 수신 : 동기화 서버**
	1. Redis의 주문 요청 데이터를 그대로 Postgre에 저장
4. 매칭 서버 (이벤트 발생 여부와 무관하게 계속 도는 상태)
	1. Kafka로부터 체결가 틱데이터를 수신받고, 해당 틱데이터마다 위에서 정의한 매칭 프로세스를 수행
	2. 매칭 프로세스 수행 결과에 따라 Redis에 저장된 주문서를 업데이트
	3. 매칭 프로세스 수행 결과의 체결 상태에 따라, Kafka에 상태 업데이트 이벤트 발행
	   (traceId에 대해 상태 변경 이벤트 발행)
5. **상태 업데이트 이벤트 수신 : 동기화 서버**
	1. 업데이트가 가능한 상태라면, 상태 업데이트 진행
		- 해당 traceId로 주문 요청 데이터가 postgre에 존재하고, 유효한 상태 업데이트라면 업데이트
		- `PENDING` -> `PARTIALLY_FILLED`
		- `PENDING` -> `FILLED`
		- `PENDING` -> `CANCELED`
		- `PARTIALLY_FILLED` -> `FILLED`
		- `PARTIALLY_FILLED` -> `PARTIALLY_CANCELED`
6. 정합성 체크 배치
	1. 매일 장 마감 이후에 실행
	2. Redis 주문 데이터를 기준으로 Postgre 동기화 처리
		- 어긋난 데이터, 누락된 데이터, 장마감까지 체결되지 않은 데이터 등을 모두 업데이트
	3. Redis 의 모든 주문 데이터 clear

## 2) 체결가 구독 종목 관리 플로우

1. 30초마다 전체 플로우 트리거
2. 구독 후보 종목 수집
	1. 미체결 주문 종목 : Redis 에 존재하는 미체결 주문 종목을 수집한다.
	2. 사용자 조회 종목 : 사용자가 현재가 페이지에 접속했을때의 종목 정보를 Redis에서 수집한다.
	3. 거래량 상위 20위 : [한투 거래량순위 Rest API](https://apiportal.koreainvestment.com/apiservice-apiservice?/uapi/domestic-stock/v1/quotations/volume-rank) 를 통해 상위 20위를 가져온다.
3. 현재 구독중인 종목 로드
	1. Redis에서 현재 구독중인 종목 리스트를 가져온다.
4. 구독 종목 리스트 업데이트
	1. 2번과 3번을 조합하여, 새로운 구독 종목 리스트를 구성한다.
	2. 이를 Redis에 저장한다. (기존 구독 종목 리스트는 그대로 둔다.)
5. 구독 종목 변경 이벤트 발행
	1. Kafka에 구독 종목 변경 이벤트를 발행한다.
6. Stock Data Gateway: 이벤트 수신
	1. 5번 이벤트를 수신받아, Redis의 새로운 구독 종목 리스트를 가져온다.
	2. 새로운 구독 종목 리스트에 따라 구독 종목을 변경한다.
	3. Redis의 기존 구독 종목 리스트를 새로운 구독 종목 리스트로 업데이트하고, 새로운 구독 종목 리스트는 제거한다.
