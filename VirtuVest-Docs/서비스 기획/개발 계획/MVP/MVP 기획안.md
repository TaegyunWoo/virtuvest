# VirtuVest MVP 기획안

## 1. MVP 개요

### 1.1 MVP 정의
VirtuVest의 핵심 가치인 **"무위험 학습 환경에서의 실시간 주식 투자 시뮬레이션"**을 검증하기 위한 최소 기능 집합

### 1.2 MVP 목표
- 실시간 주식 데이터를 활용한 가상 투자 경험 제공
- 포트폴리오 구성 및 수익률 추적 기능 검증
- 사용자 피드백 수집을 통한 서비스 방향성 검증

### 1.3 MVP 기간
- **개발 기간**: 5~6주 (2026년 3월 ~ 5월)
- **런칭 목표**: 2026년 5월 말

## 2. MVP 핵심 기능

### 2.1 사용자 관리
| 기능 | 설명 | 우선순위 |
|------|------|----------|
| 회원가입 | 이메일 기반 회원가입 | 필수 |
| 로그인/로그아웃 | JWT 기반 인증 | 필수 |

> **v2 예정**: 프로필 관리 (닉네임, 투자 성향 설정)

### 2.2 가상 계좌 및 자본
| 기능 | 설명 | 우선순위 |
|------|------|----------|
| 가상 계좌 생성 | 회원가입 시 자동 생성 | 필수 |
| 기본 자본 지급 | 1억 원 가상 자본 | 필수 |
| 잔고 조회 | 현금 잔고 확인 | 필수 |
| 자동 재충전 | 매주 500만 원 자동 지급 | 필수 |

### 2.3 주식 데이터
| 기능 | 설명 | 우선순위 |
|------|------|----------|
| 실시간 시세 조회 | 한국투자증권 API 연동 (현재가) | 필수 |
| 종목 검색 | 종목명/코드로 검색 | 필수 |
| 기본 기업 정보 | 종목명, 코드, 업종, 현재가 | 필수 |
| 일봉 차트 | 종목별 일봉 데이터 (최근 30일) | 필수 |

> **v2 예정**: 기술적 지표 (이동평균선, RSI, MACD 등)

### 2.4 거래 기능
| 기능 | 설명 | 우선순위 |
|------|------|----------|
| 매수 주문 | 시장가 매수 (즉시 체결) | 필수 |
| 매도 주문 | 시장가 매도 (즉시 체결) | 필수 |
| 거래 내역 조회 | 매수/매도 이력 | 필수 |

> **v2 예정**: 지정가 주문, 미체결 주문 관리, 주문 취소

### 2.5 포트폴리오 관리
| 기능 | 설명 | 우선순위 |
|------|------|----------|
| 보유 종목 조회 | 현재 보유 주식 목록 | 필수 |
| 수익률 계산 | 총 수익률 (평가금액 - 투자원금) | 필수 |
| 목표 수익액 설정 | 사용자 지정 목표 수익액 | 필수 |
| 목표 달성 알림 | 목표 수익액 달성 시 표시 | 필수 |

> **v2 예정**: 개별 종목 수익률, 포트폴리오 구성비 시각화

### 2.6 대시보드
| 기능 | 설명 | 우선순위 |
|------|------|----------|
| 총 자산 현황 | 평가금액 + 현금 | 필수 |
| 총 수익률 | (평가금액 - 초기자본) / 초기자본 | 필수 |

> **v2 예정**: 일별 수익률 그래프, KOSPI/KOSDAQ 지수

## 5. 데이터베이스 설계 (MVP)

### 5.1 핵심 테이블 (PostgreSQL)

```sql
-- 사용자
users (
  id, email, password_hash, nickname, 
  created_at, updated_at
)

-- 가상 계좌
virtual_accounts (
  id, user_id, balance, 
  created_at, updated_at
)

-- 보유 종목
holdings (
  id, account_id, stock_code, 
  quantity, avg_price, created_at, updated_at
)

-- 거래 내역
transactions (
  id, account_id, stock_code, 
  type (BUY/SELL), quantity, price, 
  total_amount, created_at
)

-- 주식 기본 정보
stocks (
  code, name, market (KOSPI/KOSDAQ), 
  sector, created_at, updated_at
)
```

### 5.2 시계열 테이블 (TimescaleDB)

```sql
-- 실시간/분 단위 시세 (Hypertable)
stock_prices (
  time            TIMESTAMPTZ NOT NULL,
  stock_code      VARCHAR(6) NOT NULL,
  open            DECIMAL(18,2),
  high            DECIMAL(18,2),
  low             DECIMAL(18,2),
  close           DECIMAL(18,2),
  volume          BIGINT,
  PRIMARY KEY (time, stock_code)
);

-- 일봉 데이터 (Hypertable)
daily_stock_prices (
  time            TIMESTAMPTZ NOT NULL,
  stock_code      VARCHAR(6) NOT NULL,
  open            DECIMAL(18,2),
  high            DECIMAL(18,2),
  low             DECIMAL(18,2),
  close           DECIMAL(18,2),
  volume          BIGINT,
  trading_value   DECIMAL(20,2),  -- 거래대금
  PRIMARY KEY (time, stock_code)
);

-- 시장 지수 (Hypertable)
market_indices (
  time            TIMESTAMPTZ NOT NULL,
  index_code      VARCHAR(10) NOT NULL,  -- KOSPI, KOSDAQ
  value           DECIMAL(18,2),
  change          DECIMAL(18,2),
  change_rate     DECIMAL(8,4),
  PRIMARY KEY (time, index_code)
);

-- 자동 압축 정책 (7일 후)
SELECT add_compression_policy('stock_prices', INTERVAL '7 days');
SELECT add_compression_policy('daily_stock_prices', INTERVAL '30 days');
SELECT add_compression_policy('market_indices', INTERVAL '7 days');

-- 데이터 보존 정책
SELECT add_retention_policy('stock_prices', INTERVAL '3 months');  -- 분봉: 3개월
SELECT add_retention_policy('daily_stock_prices', INTERVAL '5 years');  -- 일봉: 5년
SELECT add_retention_policy('market_indices', INTERVAL '1 year');
```

## 6. API 설계 (MVP)

### 6.1 인증 API
```
POST   /api/auth/signup        - 회원가입
POST   /api/auth/login         - 로그인
POST   /api/auth/logout        - 로그아웃
GET    /api/auth/me            - 내 정보 조회
```

### 6.2 계좌 API
```
GET    /api/account            - 계좌 정보 조회
GET    /api/account/balance   - 잔고 조회
```

### 6.3 주식 API
```
GET    /api/stocks             - 종목 목록 (검색)
GET    /api/stocks/:code       - 종목 상세 정보 (현재가 포함)
GET    /api/stocks/:code/daily - 일봉 차트 데이터 (최근 30일)
```

### 6.4 거래 API
```
POST   /api/trades/buy         - 시장가 매수
POST   /api/trades/sell        - 시장가 매도
GET    /api/trades             - 거래 내역
```

### 6.5 포트폴리오 API
```
GET    /api/portfolio          - 포트폴리오 현황 (보유 종목, 총 자산, 수익률)
```

## 12. 비즈니스 로직 상세

### 12.1 거래 규칙
| 항목 | 정책 |
|------|------|
| 최소 거래 단위 | 1주 |
| 소수점 주식 | 불가능 (정수 주만) |
| 거래 가능 시간 | 09:00 ~ 15:30 (장 운영 시간) |
| 장외 시간 거래 | 불가능 (에러 메시지 표시) |

### 12.2 수수료 정책
| 항목 | 정책 |
|------|------|
| 매수/매도 수수료 | 0.015% (가상, 실제 차감 없음) |
| 수수료 반올림 | 소수점 첫째자리 반올림 |
| 최소 수수료 | 없음 |

### 12.3 체결 로직
- **시장가 체결**: 현재가 기준 즉시 체결
- **호가 창**: MVP에서는 단일 가격(현재가)만 사용
- **동시성 제어**: 계좌 단위 락킹으로 순차 처리

### 12.4 예외 처리
| 상황 | 처리 방식 |
|------|----------|
| 잔고 부족 | "잔고가 부족합니다" 에러 반환 |
| 보유 수량 초과 매도 | "보유 수량을 초과했습니다" 에러 반환 |
| 장외 시간 거래 | "장 운영 시간(09:00~15:30)에만 거래 가능합니다" |
| 존재하지 않는 종목 | "존재하지 않는 종목입니다" 에러 반환 |

### 12.5 재충전 정책
| 항목 | 정책 |
|------|------|
| 재충전 주기 | 매주 월요일 00:00 |
| 재충전 금액 | 500만 원 |
| 최대 잔고 제한 | 없음 |

## 13. 보안 및 규정 준수

### 13.1 인증 보안
| 항목 | 정책 |
|------|------|
| 비밀번호 최소 길이 | 8자 이상 |
| 비밀번호 복잡성 | 영문 + 숫자 + 특수문자 조합 권장 |
| 비밀번호 해싱 | BCrypt (strength 10) |
| 로그인 실패 제한 | 5회 실패 시 15분 잠금 |

### 13.2 JWT 토큰 전략
| 항목 | 정책 |
|------|------|
| Access Token 만료 | 1시간 |
| Refresh Token 만료 | 7일 |
| 토큰 저장 위치 | Access: Memory, Refresh: httpOnly Cookie |
| 토큰 갱신 | Refresh Token으로 재발급 |

### 13.3 개인정보 처리
| 항목 | 내용 |

## 18. 알고리즘 설계 간소화 (Full → MVP)

### 18.1 개요

본 섹션은 [시스템 알고리즘 설계](obsidian://open?vault=VirtuVest-Docs&file=%EC%84%9C%EB%B9%84%EC%8A%A4%20%EA%B8%B0%ED%9A%8D%2F%EC%A3%BC%EC%8B%9D%20%EB%8D%B0%EC%9D%B4%ED%84%B0%20%EA%B4%80%EB%A6%AC%2F%EC%8B%9C%EC%8A%A4%ED%85%9C%20%EC%95%8C%EA%B3%A0%EB%A6%AC%EC%A6%98%20%EC%84%A4%EA%B3%84) 문서에서 정의한 정식 알고리즘을 MVP 단계에서 어떻게 간소화하여 구현하는지 정리한다.

| 알고리즘 | 정식 버전 핵심 | MVP 간소화 방향 |
|---------|-------------|---------------|
| 주문 매칭 프로세스 | 시장가 + 지정가 FIFO 큐, 부분체결, 실시간 틱 기반 매칭 | 시장가 전용, 5초 폴링 기반 즉시 전량체결 |
| 체결가 구독 종목 관리 | 우선순위 기반 최대 250종목, 다중 세션, Fallback REST | 단일 세션 고정 종목, REST 폴링 전용 |
| 시스템 플로우 | Redis 주도 + Kafka 이벤트 + 동기화 서버 + 정합성 배치 | PostgreSQL 직접 처리, Kafka 미사용, 동기화 서버 불필요 |

> **원칙**: MVP에서 제거·간소화한 모든 요소는 v2 이후 정식 알고리즘으로 전환할 수 있도록 데이터 모델과 인터페이스를 설계한다.

### 18.2 주문 매칭 프로세스 간소화

#### 18.2.1 정식 알고리즘 요약

정식 버전은 한투 실시간 체결가 API에서 틱 데이터를 수신하고, 시장가 큐(`Deque`)와 지정가 큐(`Map<호가, 큐>`)를 호가별 FIFO로 관리한다. 매수는 시장가 우선 → 체결가 이상 최고호가, 매도는 시장가 우선 → 체결가 이하 최저호가 순으로 매칭하며, `min(API 체결 수량, 주문 수량)`만큼 체결 처리한다. 체결 결과에 따라 `EXECUTED`, `PARTIALLY_FILLED` 상태를 구분하고 잔량을 이월한다.

#### 18.2.2 MVP 간소화 버전

| 구성 요소 | 정식 버전 | MVP | 사유 |
|----------|----------|-----|------|
| 주문 유형 | 시장가 + 지정가 | 시장가만 지원 | 지정가는 호가별 큐 관리 필요 → v2 |
| 주문 큐 구조 | `Map<호가, 큐>` + `Deque` | 단순 FIFO 큐 (시장가 전용) | 호가 분류 불필요 |
| 체결 데이터 소스 | 실시간 WebSocket 틱 데이터 | REST API 5초 간격 폴링 | WebSocket 구독 관리 복잡도 제거 |
| 체결 수량 처리 | `min(API 체결 수량, 주문 수량)` 부분체결 | 전량 즉시체결 (현재가 기준) | 부분체결 상태 관리 제거 |
| 주문 상태 | `PENDING` → `PARTIALLY_FILLED` → `EXECUTED` → `CANCELED` | `PENDING` → `EXECUTED` | 중간 상태 불필요 |
| 동시성 제어 | 종목별 락킹 | 계좌 단위 락킹 | Section 12.3 기준 유지 |

#### 18.2.3 MVP 체결 플로우

1. 사용자가 시장가 매수/매도 주문 요청
2. 주문 서버가 잔고 검증 후 `PENDING` 상태로 PostgreSQL에 저장
3. 5초 간격 폴링으로 해당 종목의 현재가를 조회
4. 현재가 기준으로 전량 즉시체결 처리
5. 주문 상태를 `EXECUTED`로 업데이트, 포트폴리오 반영
6. 거래 내역 기록

#### 18.2.4 v2 확장 경로

- 지정가 주문 도입 → 호가별 `Map<호가, 큐>` 구조 추가
- 실시간 WebSocket 틱 데이터 수신으로 전환
- `PARTIALLY_FILLED` 상태 및 부분체결 로직 구현
- 종목별 락킹으로 동시성 제어 세분화

### 18.3 체결가 구독 종목 관리 간소화

#### 18.3.1 정식 알고리즘 요약

정식 버전은 미체결 주문 종목 → 사용자 조회 종목 → 거래량 상위 20위 순으로 우선순위를 두고, 세션당 50종목 × 최대 5세션(총 250종목)을 WebSocket으로 구독한다. 30초마다 구독 후보를 재수집하며, 250종목 초과 시 REST API 0.2초 폴링으로 Fallback 처리한다. 구독 종목 리스트는 Redis에서 세션별로 관리하고, Kafka 이벤트를 통해 Stock Data Gateway와 동기화한다.

#### 18.3.2 MVP 간소화 버전

| 구성 요소 | 정식 버전 | MVP | 사유 |
|----------|----------|-----|------|
| 구독 방식 | WebSocket 실시간 구독 | REST API 5초 간격 폴링 | WebSocket 세션 관리 불필요 |
| 세션 관리 | 최대 5세션 × 50종목 | 세션 관리 없음 | 폴링 방식이므로 세션 개념 불필요 |
| 종목 선정 | 우선순위 기반 동적 선정 (최대 250종목) | 사용자 보유 종목 + 현재 조회 종목만 | 동적 우선순위 로직 제거 |
| 갱신 주기 | 30초마다 구독 목록 재수집 | 5초 간격 폴링 시 대상 종목 결정 | 별도 갱신 프로세스 불필요 |
| Fallback | REST API 0.2초 폴링 | REST API가 기본 방식 | Fallback 분기 불필요 |
| 저장소 | Redis 세션별 관리 + Kafka 이벤트 | 폴링 시 직접 조회 | Redis 구독 관리, Kafka 이벤트 제거 |

#### 18.3.3 MVP 데이터 수집 플로우

1. 5초 간격 스케줄러가 현재가 조회 대상 종목 결정 (보유 종목 + 사용자 조회 종목)
2. 한투 REST API로 해당 종목들의 현재가 일괄 조회
3. 조회 결과를 Redis 캐시에 저장 (캐싱 전용)
4. 클라이언트 요청 시 캐시된 현재가 반환

#### 18.3.4 v2 확장 경로

- WebSocket 실시간 구독 도입 (세션당 50종목)
- 우선순위 기반 동적 종목 선정 로직 구현
- 다중 세션 관리 (최대 5세션, 250종목)
- 30초 주기 구독 목록 재수집 프로세스 추가
- 250종목 초과 시 REST Fallback 분기 구현

### 18.4 시스템 플로우 간소화

#### 18.4.1 정식 시스템 플로우 요약

정식 버전의 주문 시스템은 Redis를 주 저장소로 사용하고, Kafka를 통해 이벤트 기반으로 동작한다. 주문 서버가 Redis에 저장 후 Kafka로 이벤트를 발행하면, 동기화 서버가 수신하여 PostgreSQL에 백업한다. 매칭 서버는 Kafka 틱 데이터를 수신해 매칭을 수행하고, 결과를 Redis에 반영한 뒤 상태 변경 이벤트를 발행한다. 체결가 구독 관리는 30초마다 트리거되어 Redis + Kafka를 통해 Stock Data Gateway와 동기화한다. 장 마감 후 정합성 배치가 Redis → PostgreSQL 동기화를 수행한다.

#### 18.4.2 MVP 간소화 버전

| 구성 요소 | 정식 버전 | MVP | 사유 |
|----------|----------|-----|------|
| 주 저장소 | Redis (RDB는 백업) | PostgreSQL 직접 처리 | 단일 저장소로 복잡도 감소 |
| 메시지 브로커 | Kafka 이벤트 기반 | 미사용 | 동기화 서버 불필요로 이벤트 발행 불필요 |
| 동기화 서버 | Redis → PostgreSQL 동기화 전담 | 미사용 | PostgreSQL 직접 처리로 동기화 불필요 |
| 매칭 서버 | 독립 프로세스, Kafka 틱 수신 | 주문 서버 내 매칭 로직 통합 | 별도 서버 불필요 (5초 폴링 기반) |
| Redis 역할 | 주문 데이터 주 저장소 + 구독 관리 | 캐싱 전용 (현재가, 세션 등) | Section 4.2 기준 유지 |
| 체결가 구독 관리 | 30초 트리거 + Kafka + Stock Data Gateway | 5초 폴링 스케줄러 | 구독 관리 프로세스 전체 제거 |
| 상태 전이 | `PENDING` → `PARTIALLY_FILLED` → `FILLED` → `CANCELED` | `PENDING` → `EXECUTED` | 중간 상태 제거 |
| 정합성 배치 | 장 마감 후 Redis → PostgreSQL 동기화 | 미사용 | 단일 저장소(PostgreSQL)로 정합성 이슈 없음 |

#### 18.4.3 MVP 시스템 플로우

1. **주문 요청**: 클라이언트 → 주문 서버로 시장가 매수/매도 요청
2. **주문 처리**: 주문 서버가 잔고 검증 후 PostgreSQL에 `PENDING` 상태로 저장 (계좌 단위 락킹)
3. **현재가 조회**: 5초 간격 폴링 스케줄러가 한투 REST API로 보유 종목 + 조회 종목 현재가 조회
4. **체결 처리**: 현재가 기준 전량 즉시체결 → PostgreSQL 주문 상태 `EXECUTED` 업데이트 + 포트폴리오 반영
5. **결과 반환**: 체결 결과를 클라이언트에 반환

> 현재가 데이터는 Redis에 캐싱하여 클라이언트의 시세 조회 요청에 활용한다 (Section 4.2).

#### 18.4.4 v2 확장 경로

- Redis를 주문 데이터 주 저장소로 전환
- Kafka 도입으로 이벤트 기반 아키텍처 구현
- 동기화 서버 분리 (Redis → PostgreSQL 비동기 백업)
- 매칭 서버 독립 프로세스로 분리
- 장 마감 정합성 배치 프로세스 구현
